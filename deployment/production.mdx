This guide covers production deployment considerations for The Fork application, including security, performance, monitoring, and infrastructure best practices.

<Warning>
  The Fork deals with sensitive personal conversations. Production deployments require careful consideration of ethics, safety, moderation, and user privacy.
</Warning>

## Pre-Deployment Checklist

Before deploying to production:

- [ ] Ethics review completed
- [ ] Content moderation strategy defined
- [ ] Crisis resources integrated
- [ ] Privacy policy created
- [ ] Terms of service defined
- [ ] Rate limiting configured
- [ ] Monitoring and alerting set up
- [ ] Backup strategy implemented
- [ ] Incident response plan created
- [ ] Security audit completed

## Security Configuration

### CORS Settings

<Warning>
  Never use `CORS_ORIGINS="*"` in production. This allows any website to make requests to your API.
</Warning>

Restrict CORS to your frontend domain:

```bash backend/.env
# Single origin
CORS_ORIGINS="https://yourdomain.com"

# Multiple origins (comma-separated)
CORS_ORIGINS="https://yourdomain.com,https://www.yourdomain.com"
```

### TLS/SSL Configuration

Always use HTTPS in production. Use a reverse proxy like Nginx:

```nginx nginx.conf
server {
    listen 80;
    server_name yourdomain.com;
    return 301 https://$server_name$request_uri;
}

server {
    listen 443 ssl http2;
    server_name yourdomain.com;

    ssl_certificate /etc/nginx/ssl/cert.pem;
    ssl_certificate_key /etc/nginx/ssl/key.pem;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;

    # Frontend
    location / {
        proxy_pass http://frontend:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # Backend API
    location /api/ {
        proxy_pass http://backend:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

For Let's Encrypt SSL certificates:

```bash
# Install certbot
sudo apt-get install certbot python3-certbot-nginx

# Obtain certificate
sudo certbot --nginx -d yourdomain.com -d www.yourdomain.com

# Auto-renewal is configured automatically
```

## Rate Limiting

Protect your API and LLM costs with rate limiting:

### Nginx Rate Limiting

```nginx nginx.conf
http {
    # Define rate limit zones
    limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/m;
    limit_req_zone $binary_remote_addr zone=chat_limit:10m rate=5r/m;

    server {
        # General API endpoints
        location /api/ {
            limit_req zone=api_limit burst=20 nodelay;
            proxy_pass http://backend:8000;
        }

        # Chat endpoint (more restrictive)
        location /api/chat {
            limit_req zone=chat_limit burst=3 nodelay;
            proxy_pass http://backend:8000;
        }
    }
}
```

### Application-Level Rate Limiting

Add to `backend/server.py`:

```python
from slowapi import Limiter, _rate_limit_exceeded_handler
from slowapi.util import get_remote_address
from slowapi.errors import RateLimitExceeded

limiter = Limiter(key_func=get_remote_address)
app.state.limiter = limiter
app.add_exception_handler(RateLimitExceeded, _rate_limit_exceeded_handler)

@api_router.post("/chat")
@limiter.limit("5/minute")
async def chat_endpoint(request: Request, chat_request: ChatRequest):
    # ... existing code
```

Install dependency:

```bash
pip install slowapi
```

## Database Configuration

### MongoDB Production Setup

<Steps>
  <Step title="Enable authentication">
    Create admin and application users:
    
    ```javascript
    use admin
    db.createUser({
      user: "admin",
      pwd: "secure_admin_password",
      roles: [{ role: "userAdminAnyDatabase", db: "admin" }]
    })

    use fork_database
    db.createUser({
      user: "fork_app",
      pwd: "secure_app_password",
      roles: [{ role: "readWrite", db: "fork_database" }]
    })
    ```
  </Step>

  <Step title="Configure replica set">
    For high availability:
    
    ```bash
    # Initialize replica set
    mongosh --eval 'rs.initiate()'

    # Add members
    rs.add("mongodb-2:27017")
    rs.add("mongodb-3:27017")
    ```
  </Step>

  <Step title="Enable encryption at rest">
    ```yaml docker-compose.yml
    mongo:
      command: mongod --enableEncryption --encryptionKeyFile /etc/mongodb-keyfile
      volumes:
        - ./mongodb-keyfile:/etc/mongodb-keyfile:ro
    ```
  </Step>

  <Step title="Configure backups">
    ```bash
    # Automated backup script
    #!/bin/bash
    DATE=$(date +%Y%m%d_%H%M%S)
    mongodump --uri="mongodb://user:pass@host/fork_database" \
      --out="/backups/fork_${DATE}"

    # Compress and upload to S3
    tar -czf "/backups/fork_${DATE}.tar.gz" "/backups/fork_${DATE}"
    aws s3 cp "/backups/fork_${DATE}.tar.gz" s3://your-bucket/backups/

    # Cleanup old backups (keep last 30 days)
    find /backups -name "fork_*.tar.gz" -mtime +30 -delete
    ```
    
    Add to crontab:
    ```bash
    # Daily backup at 2 AM
    0 2 * * * /path/to/backup-script.sh
    ```
  </Step>
</Steps>

<Note>
  While MongoDB is included in the template, The Fork application is stateless and doesn't persist conversations. You may choose to disable MongoDB if not using the optional template endpoints.
</Note>

### Health Checks and Uptime Monitoring

```python backend/server.py
from fastapi import status

@api_router.get(
    "/health",
    status_code=status.HTTP_200_OK,
    tags=["health"]
)
async def health_check():
    """Comprehensive health check endpoint"""
    health_status = {
        "status": "healthy",
        "timestamp": datetime.utcnow().isoformat(),
        "checks": {}
    }

    # Check MongoDB
    try:
        await db.command('ping')
        health_status["checks"]["mongodb"] = "ok"
    except Exception as e:
        health_status["checks"]["mongodb"] = f"error: {str(e)}"
        health_status["status"] = "degraded"

    # Check LLM API
    try:
        llm_key = os.getenv("EMERGENT_LLM_KEY")
        health_status["checks"]["llm_key"] = "configured" if llm_key else "missing"
        if not llm_key:
            health_status["status"] = "degraded"
    except Exception as e:
        health_status["checks"]["llm_key"] = f"error: {str(e)}"
        health_status["status"] = "degraded"

    return health_status
```

Configure uptime monitoring (e.g., UptimeRobot, Pingdom, StatusCake):
- Monitor: https://yourdomain.com/api/health
- Frequency: Every 5 minutes
- Alert: Email/SMS on failure

### Structured Logging

```python backend/server.py

from datetime import datetime

class StructuredLogger:
    def __init__(self, name):
        self.logger = logging.getLogger(name)
        self.logger.setLevel(logging.INFO)
```

## Safety and Ethics in Production

### Content Moderation

The Fork includes built-in safety checks in `backend/server.py`:

1. **Self-harm detection** - Returns crisis resources instead of engaging
2. **Hate speech detection** - Refuses and redirects conversation
3. **System prompt guardrails** - Forbids slurs and abusive content

### User Privacy

<Warning>
  The Fork is designed to be ephemeral - conversations are NOT persisted. Maintain this privacy-first approach in production.
</Warning>

Privacy best practices:

1. **No conversation logging** - Do not store user messages or LLM responses
2. **Minimal metadata** - Only log session IDs, timestamps, and technical metrics
3. **Data retention policy** - Clear logs older than 30 days
4. **GDPR compliance** - Provide data deletion on request
5. **Transparent privacy policy** - Clearly communicate data practices

```python
# Example: Sanitize logs
def sanitize_for_logging(data: dict) -> dict:
    """Remove PII from log data"""
    sanitized = data.copy()
    # Remove message content, keep only metadata
    if "messages" in sanitized:
        sanitized["message_count"] = len(sanitized["messages"])
        del sanitized["messages"]
    if "forkStatement" in sanitized:
        del sanitized["forkStatement"]
    return sanitized
```

## Cost Optimization

### LLM API Costs

The primary cost is Emergent LLM API calls:

1. **Implement rate limiting** (see above) - 5 requests/minute per user
2. **Set maximum tokens** - Limit response length
3. **Monitor usage** - Track API calls and costs
4. **Implement quotas** - Per-user daily limits

```python
from emergentintegrations.llm.chat import LlmChat

# Configure max tokens
llm_chat = LlmChat(
    provider="openai",
    model="gpt-5.2",
    max_tokens=500,  # Limit response length
    temperature=0.7
)
```

### Infrastructure Costs

Optimize cloud spending:

1. **Use spot instances** for non-critical workloads
2. **Auto-scaling** - Scale down during low traffic
3. **Reserved instances** for predictable base load
4. **CDN for static assets** (CloudFront, Cloudflare)
5. **Optimize Docker images** - Smaller images = faster deploys, lower storage

## Disaster Recovery

### Backup Strategy

1. **MongoDB backups** - Daily automated backups to S3
2. **Configuration backups** - Version control all configs
3. **Docker image registry** - Retain last 10 versions
4. **Secrets backup** - Encrypted backup of secrets in secure location

### Incident Response Plan

<Steps>
  <Step title="Incident Detection">
    - Automated alerts via monitoring
    - User reports via support channels
    - Health check failures
  </Step>

  <Step title="Assessment">
    - Severity level (P0-P4)
    - Impact scope (users affected)
    - Root cause analysis
  </Step>

  <Step title="Response">
    - P0 (Critical): Immediate rollback or failover
    - P1 (High): Fix within 1 hour
    - P2 (Medium): Fix within 4 hours
    - P3 (Low): Fix within 24 hours
  </Step>

  <Step title="Resolution">
    - Apply fix
    - Verify resolution
    - Post-mortem documentation
  </Step>

  <Step title="Prevention">
    - Update runbooks
    - Add monitoring/alerts
    - Improve tests
  </Step>
</Steps>
